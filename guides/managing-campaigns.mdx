Campaigns are the core mechanism for distributing scraped usernames to Virtual Assistants (VAs). This guide covers creating campaigns, viewing history, and understanding campaign status.

## What is a Campaign?

A campaign represents a batch distribution of usernames to VA tables in Airtable. When you create a campaign:

1. Usernames are pulled from the `global_usernames` pool (where `used = false`)
2. They're distributed across VA tables based on your configuration
3. Assignments are recorded in the `daily_assignments` table
4. Campaign status is tracked in the `campaigns` table

## Campaign Data Structure

```typescript
interface Campaign {
  campaign_id: string;      // UUID - unique identifier
  campaign_date: string;    // ISO timestamp of creation
  total_assigned: number;   // Total usernames distributed
  status: boolean;          // true = success, false = failed
  created_at: string;       // ISO timestamp
  base_id: string;          // Multi-tenant isolation key
}
```

## How Campaigns Are Created

Campaigns are created through the scraping and distribution workflow:

### 1. Username Scraping

First, usernames must be scraped and added to the `global_usernames` table:

```tsx
export async function addUsernamesToJob(
  jobId: string,
  baseId: string,
  usernames: Array<{
    id: string;
    username: string;
    full_name?: string;
  }>
): Promise<boolean> {
  const { error } = await supabase
    .from('global_usernames')
    .insert(
      usernames.map(u => ({
        id: u.id,
        username: u.username,
        full_name: u.full_name || null,
        job_id: jobId,
        used: false,        // Available for assignment
        base_id: baseId     // Multi-tenant isolation
      }))
    );

  return !error;
}
```

### 2. Assignment Distribution

Usernames are then distributed to VA tables:

```tsx
export async function createJobAssignments(
  jobId: string,
  campaignId: string,
  baseId: string,
  assignments: Array<{
    id: string;
    username: string;
    full_name?: string;
    va_table_number: number;
    position: number;
  }>
): Promise<boolean> {
  const { error } = await supabase
    .from('daily_assignments')
    .insert(
      assignments.map(a => ({
        campaign_id: campaignId,
        id: a.id,
        username: a.username,
        full_name: a.full_name || null,
        va_table_number: a.va_table_number,
        position: a.position,
        job_id: jobId,
        status: 'pending',
        base_id: baseId
      }))
    );

  return !error;
}
```

### 3. Campaign Record Creation

Once distribution is complete, the campaign is recorded:

```tsx
await supabase
  .from('campaigns')
  .insert({
    campaign_id: uuid(),
    campaign_date: new Date().toISOString(),
    total_assigned: assignments.length,
    status: true,  // Success
    base_id: baseId
  });
```

## Multi-Tenant Isolation

Campaigns are isolated by `base_id` to ensure data separation:

```tsx
export function CampaignsTable() {
  const { baseId, isLoading: isLoadingBase } = useBase()
  
  const fetchCampaigns = async () => {
    if (!baseId) return;
    
    // Create a Supabase client with the base_id context
    const supabaseWithContext = createSupabaseClientWithContext(baseId)
    
    // Query with explicit base_id filter
    const { data, error } = await supabaseWithContext
      .from('campaigns')
      .select('campaign_id, campaign_date, total_assigned, status, created_at')
      .eq('base_id', baseId)  // Belt-and-suspenders approach
      .order('campaign_date', { ascending: false })
    
    setCampaigns(data || [])
  }
}
```

This ensures that when viewing campaigns:
- You only see campaigns for the current active job
- Switching jobs shows different campaign history
- No cross-contamination between different influencer jobs

## Job Statistics

Track overall job progress with statistics:

```tsx
export async function getJobStatisticsByBaseId(baseId: string): Promise<JobStatistics | null> {
  // Get available usernames count (used = false)
  const { count: availableUsernames } = await supabase
    .from('global_usernames')
    .select('*', { count: 'exact', head: true })
    .eq('base_id', baseId)
    .eq('used', false);

  // Get used usernames count
  const { count: usedUsernames } = await supabase
    .from('global_usernames')
    .select('*', { count: 'exact', head: true })
    .eq('base_id', baseId)
    .eq('used', true);

  // Get total usernames count
  const { count: totalUsernames } = await supabase
    .from('global_usernames')
    .select('*', { count: 'exact', head: true })
    .eq('base_id', baseId);

  // Get assignments count
  const { count: totalAssignments } = await supabase
    .from('daily_assignments')
    .select('*', { count: 'exact', head: true })
    .eq('base_id', baseId);

  return {
    totalUsernames: totalUsernames || 0,
    usedUsernames: usedUsernames || 0,
    availableUsernames: availableUsernames || 0,
    totalAssignments: totalAssignments || 0,
  };
}
```

### Statistics Display

  totalAssignments: number;     // Total assignments across all campaigns
  totalScrapeResults: number;   // Total scrape operations

## Assignment Workflow

Here's the complete workflow from scraping to assignment:

<Steps>
  <Step title="Scrape profiles">
    Use the scraping functionality to pull follower/engagement data from the social platform.
    
    Scraped usernames are stored in `global_usernames` with `used = false`.
  </Step>

  <Step title="Check available count">
    Query how many unused usernames are available:
    
    ```tsx
    const { count } = await supabase
      .from('global_usernames')
      .select('*', { count: 'exact', head: true })
      .eq('base_id', baseId)
      .eq('used', false);
    ```
  </Step>

  <Step title="Create campaign">
    Generate a new campaign ID and prepare assignments:
    
    ```tsx
    const campaignId = uuid();
    const target = 14400; // Daily selection target
    const numVAs = 10;    // Number of VA tables
    const perVA = target / numVAs; // 1,440 per VA
    ```
  </Step>

  <Step title="Distribute usernames">
    Assign usernames to each VA table:
    
    - Pull `perVA` unused usernames from `global_usernames`
    - Create assignment records in `daily_assignments`
    - Mark usernames as `used = true`
    - Set `va_table_number` for each assignment
  </Step>

  <Step title="Sync to Airtable">
    Push assignments to the corresponding VA tables in Airtable:
    
    - Table names: `VA 1`, `VA 2`, etc.
    - Fields: Username, Full Name, Status, Position
    - Batch create records via Airtable API
  </Step>

  <Step title="Record campaign">
    Save the campaign record with final status:
    
    - `status: true` if all operations succeeded
    - `status: false` if any errors occurred
    - `total_assigned` = count of assignments created
  </Step>
</Steps>

## Managing Job Status

Control job lifecycle with status updates:

```tsx
// Pause a job (stop new campaigns)
export async function pauseJob(jobId: string): Promise<boolean> {
  return updateJobStatus(jobId, 'paused');
}

// Activate a job (allow new campaigns)
export async function activateJob(jobId: string): Promise<boolean> {
  return updateJobStatus(jobId, 'active');
}

// Archive a job (soft delete)
export async function archiveJob(jobId: string): Promise<boolean> {
  return updateJobStatus(jobId, 'archived');
}
```

## Daily Selection Target

The platform supports a configurable daily target for username distribution:

```bash
# .env.local
NEXT_PUBLIC_DAILY_SELECTION_TARGET=14400
```

This determines:
- How many total usernames to distribute per campaign
- Division across VA tables (e.g., 14,400 / 10 VAs = 1,440 per VA)
- Minimum available username requirement before creating campaigns

## Common Issues

### Campaign Shows "Failed" Status

**Possible causes:**
1. Insufficient available usernames in `global_usernames`
2. Airtable API rate limit exceeded
3. Network connectivity issues
4. Database write failures

**Solution:**
- Check available username count
- Review application logs for specific errors
- Verify Airtable API credentials and rate limits
- Retry the campaign after resolving issues

### No Campaigns Appearing

**Possible causes:**
1. No campaigns created yet for this job
2. Wrong `base_id` (viewing different job)
3. Database query errors

**Solution:**
- Verify you're viewing the correct job
- Check browser console for errors
- Confirm campaigns exist in Supabase `campaigns` table

### Assignments Not Syncing to Airtable

**Possible causes:**
1. Invalid Airtable base ID
2. Missing or incorrect Airtable API key
3. Table structure mismatch
4. API rate limiting

**Solution:**
- Verify Airtable base ID matches `airtable_base_id` in job
- Check Airtable API credentials
- Ensure VA tables exist with correct field names
- Implement retry logic with exponential backoff
